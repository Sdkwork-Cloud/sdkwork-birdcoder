# Rust Backend sdkwork-specs Migration Plan

## Goal
Migrate the monolithic 32,496-line `sdkwork-birdcoder-server` Rust backend from `apps/sdkwork-birdcoder-pc/packages/` into sdkwork-specs-compliant crate architecture under `crates/`, with proper naming, layer separation, and IAM integration via `sdkwork-appbase`.

## Current State
- **5 Rust crates** all in `apps/sdkwork-birdcoder-pc/packages/` (wrong location per specs)
- **All use non-compliant names** (missing domain/suffix patterns)
- **Server lib.rs is 32,496 lines** — monolith with routes, handlers, business logic, SQLite schema, DTOs, and 8,274 lines of tests
- **iam_authority.rs (2,549 lines)** duplicates `sdkwork-appbase` IAM patterns
- **crates/ directory is empty** (just README.md)
- **No typed AppRequestContext** — handlers reparse raw headers
- **No route/service/repository separation**

## Migration Phases

### Phase 0: Workspace Preparation
1. Create all 31 crate directories under `crates/` with `Cargo.toml` and empty `src/lib.rs`
2. Update root `Cargo.toml` workspace members (keep old members temporarily)
3. Create shared `sdkwork-birdcoder-errors` crate
4. Verify: `cargo check --workspace`

### Phase 1: Domain Models and DTOs
1. Extract intelligence domain models (CodingSession*, Turn*, Event*, Artifact*, etc.) into service crate `domain/` modules
2. Extract platform domain models (Workspace*, Project*, Deployment*, Release*, Team*, etc.)
3. Extract remaining domain models (ecosystem, content, commerce, runtime, system)
4. Define port traits in each service crate `ports/` module
5. Verify: `cargo check --workspace`

### Phase 2: Repository Layer
1. Extract intelligence repository (ai_coding_session* tables, SQL queries, row mappers)
2. Extract platform repository (studio_*, ops_* tables)
3. Extract remaining repositories
4. Extract seed data and schema migration logic
5. Verify: `cargo check --workspace`

### Phase 3: Service Layer
1. Implement intelligence coding sessions service (CRUD, turn lifecycle, approvals)
2. Implement platform services (workspace, project, deployment)
3. Implement remaining services (runtime, ecosystem, content, commerce, system)
4. Verify: `cargo check --workspace`

### Phase 4: Route Crates and IAM Integration
1. Wire AppRequestContext as Axum extractor (from sdkwork-appbase)
2. Create intelligence route crate (thin handlers, paths, routes, manifest, error mapping)
3. Create remaining route crates (system, runtime, platform, content, ecosystem, commerce, iam-backend, platform-backend)
4. Wire IAM integration via sdkwork-appbase (delete iam_authority.rs)
5. Verify: `cargo check --workspace`

### Phase 5: Server and Host Assembly
1. Implement API server bootstrap (config, state, database, repositories, services, adapters, routers)
2. Implement service host (in-process container for Tauri embedding)
3. Implement Tauri host (commands, adapters, embedded server)
4. Relocate codeengine and git crates to `crates/`
5. Verify: `cargo build -p sdkwork-birdcoder-api-server`, `cargo build -p sdkwork-birdcoder-tauri-host`

### Phase 6: Test Migration and Cleanup
1. Migrate 8,274 lines of tests to appropriate crates
2. Remove old crates from `apps/sdkwork-birdcoder-pc/packages/`
3. Update root `Cargo.toml` (remove old workspace members)
4. Update TypeScript desktop host references if needed
5. Verify: `cargo test --workspace`, `pnpm run typecheck`, `pnpm run lint`

### Phase 7: Compliance Verification
1. Naming audit: no forbidden suffixes, all follow sdkwork-specs patterns
2. Structure audit: lib.rs < 150 lines, no SQL in route crates, no HTTP types in service crates
3. Request context audit: no raw header parsing, all handlers use AppRequestContext
4. Full build and test: `cargo build --workspace && cargo test --workspace && cargo clippy --workspace`
5. Architecture check: `pnpm run check:arch`

## Key Design Decisions

1. **IAM**: Delete `iam_authority.rs`, consume `sdkwork-appbase` IAM crates (`sdkwork-iam-context-service`, `sdkwork-router-iam-app-api`, `sdkwork-router-iam-backend-api`, `sdkwork-iam-directory-repository-sqlx`)

2. **Repository naming**: Uses `-repository-sqlite` (not `-sqlx`) since existing code uses `rusqlite`. Future phase can migrate to SQLx.

3. **Backward compatibility**: Old crates remain in workspace until Phase 6. Feature flags allow running old or new server during transition.

4. **Handler pattern**: Thin adapters that extract AppRequestContext, call service, map response. No business logic in handlers.

5. **Service pattern**: Receives `SessionContext` (tenant_id, user_id) — NOT HTTP types. Calls repository via trait, codeengine via trait, realtime via trait.
