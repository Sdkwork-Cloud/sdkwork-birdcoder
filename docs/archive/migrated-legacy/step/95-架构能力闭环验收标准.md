# 95 - 架构能力闭环验收标准

## 1. 目标与范围

用于判断某个 Step、某个波次或整个 BirdCoder 是否真正完成了架构能力闭环。判定对象不是“代码是否提交”，而是“架构承诺是否被实现、验证、留痕并可持续运行”。

## 2. 架构对齐

对齐 `/docs/架构/02-21` 与本目录 `00-18`、`91`、`97`、`98`、`99`。任何没有通过本标准的能力，都不能宣称已完成收敛。

## 3. 闭环最小标准

| 维度 | 必须满足 |
| --- | --- |
| 设计对齐 | 已明确对齐的架构文档、Step 编号、边界与非目标 |
| 实现落地 | 主目录改造完成，目录职责清晰，无平行主逻辑 |
| 自动化验证 | 至少 1 条契约或集成自动化验证通过 |
| 运行证据 | 至少 1 条真实执行或 Smoke 证据可回放 |
| 发布证据 | 若能力影响交付链，必须纳入发布或回滚证据 |
| 多模式一致性 | `web / desktop / server` 语义一致 |
| 多引擎一致性 | Engine 切换后 Session/Event/Artifact 语义稳定 |
| 多数据库一致性 | `sqlite / postgresql` 逻辑语义一致 |
| 文档回写 | `/docs/架构/`、`/docs/step/`、必要时 `/docs/prompts/` 已同步更新 |
| 解锁关系一致性 | 当前 Step 的完成信号、交付包、下一解锁项与 `99` 一致 |

否决条件：

- 只改代码，不回写文档
- 只有人工验证，没有自动化验证
- 能成功，不能恢复或回滚
- 多模式、多引擎、多数据库语义不一致
- 高风险动作无审计留痕

## 4. 测试与结果验证

- 每个 Step 收尾时必须用本标准逐项判断。
- 波次总验收必须检查闭环是否覆盖该波次的全部关键能力。
- 任一维度缺失都要明确标记“未闭环”，不能用主观判断代替。
- 若 `91` 未通过、`97` 未回写、`98` 未交付、`99` 解锁条件未满足，仍视为未闭环。

## 5. 检查点

- `CP95-1`：每项关键能力都能映射到闭环结论。
- `CP95-2`：闭环结论可复核、可举证、可追溯。
- `CP95-3`：`14-18` 专项能力不会被误判为“已由旧 Step 隐式覆盖”。
- `CP95-4`：未闭环能力不会进入发布链。
- `CP95-5`：闭环结论与 `99` 的完成信号、解锁项保持一致。

## 6. 执行方式

| 项 | 说明 |
| --- | --- |
| 使用时机 | 每个 Step 收尾、每个波次收尾、最终发布前 |
| 是否可并行 | 可并行准备证据，闭环结论必须统一收口 |
| 输出要求 | 闭环结论、缺口列表、补救动作、是否准入下一阶段 |
