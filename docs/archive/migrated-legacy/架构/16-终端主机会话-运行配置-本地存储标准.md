# 16-外部Terminal集成边界-运行配置映射-本地存储标准

## 1. 目标

定义 BirdCoder 与外部 Terminal 工程之间的标准接入边界：BirdCoder 只负责生成标准启动请求、记录外部引用、消费状态摘要，不承担终端会话 authority。

## 2. 标准对象

| 对象 | 关键字段 | 责任 |
| --- | --- | --- |
| `run_configuration` | `id name profile_id command cwd_mode custom_cwd group scope_type scope_id` | 本仓统一运行配置 |
| `external_terminal_launch_request` | `providerId profileId cwd workspaceId projectId source envRef` | 发往外部 Terminal 的标准启动请求 |
| `external_terminal_session_ref` | `providerSessionId providerUrl status lastSeenAt` | 外部会话引用，不是本地 authority |
| `external_terminal_evidence_ref` | `providerEvidenceId artifactRef auditRef releaseRef` | 外部证据、审计、发布引用 |

`source` 固定为 `code | studio | automation`，不再把 `terminal` 作为本仓主工作面来源。

## 3. 映射规则

- `Code / Studio` 只能发出统一 `external_terminal_launch_request`。
- 外部 Terminal 工程负责：
  - 会话创建
  - 命令执行
  - 输出流
  - 恢复语义
  - 历史归档
- BirdCoder 负责：
  - 统一启动参数
  - 工作空间 / 项目上下文映射
  - 状态摘要展示
  - 审计 / 发布回写

## 4. 本地存储规则

- 本仓允许持久化：
  - `run_configuration`
  - `external_terminal_session_ref`
  - `external_terminal_evidence_ref`
- 本仓不允许持久化：
  - `terminal_session` authority
  - `terminal_execution` authority
  - 真实输出流与缓冲
  - 命令历史全文

## 5. 治理规则

- 外部 Terminal 的审计与发布信息必须可投影为标准摘要。
- Release 与治理只消费标准摘要与 artifact 引用，不消费外部工程私有内部结构。
- 若外部 Terminal 工程版本未冻结，本仓不得提前实现深度终端交互。

## 6. 非目标

- 不在本仓实现 PTY。
- 不在本仓实现 CLI Registry。
- 不在本仓实现 Terminal Session Repository。
- 不在本仓为 Terminal 建立独立 SQLite / PostgreSQL authority 迁移链。

## 7. 评估标准

- 启动请求是否统一。
- 外部引用是否足够完成状态展示、审计、发布回写。
- 本仓是否没有继续侵入外部 Terminal 的内部 authority 领域。
