## Highlights

- Defines BirdCoder-owned Prompt and AppTemplate contracts without introducing a local Skills domain.
- Uses `skills_context` only as a prompt-composition slot populated from canonical `sdkwork-skills` installation references.
- Enforces the Skills ownership boundary in lint, release-flow, CI, and governance regression checks.

## Current architecture boundary

- `sdkwork-skills` exclusively owns Skill packages, immutable artifacts, capabilities, categories, assets, actions, and installations.
- BirdCoder consumes `@sdkwork/skills-app-sdk` with dual-token authentication and never supplies tenant, organization, or user authority from the client.
- AppTemplate inputs may store stable `skillInstallationIds`; BirdCoder does not copy Skill records, create binding tables, select a latest artifact, or assemble a second Skill runtime.

## Release guardrails

- Stop shipment if BirdCoder reintroduces a Skills table, local Skill DTO/API, raw HTTP transport, tenant selector, implicit artifact selection, or local Skill runtime authority.
- Stop shipment if `skills_context` stops representing canonical installation references or if AppTemplate inputs persist anything broader than stable `skillInstallationIds`.
- Re-issue only after the canonical Skills SDK boundary and all governance checks pass.
