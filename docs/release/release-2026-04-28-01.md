## Highlights

- Closes the multi-window programming standard by promoting `@sdkwork/birdcoder-multiwindow` from a product surface into a governed package, shell route, runtime dispatch lane, page contract, and release-backed quality gate.
- Freezes the multi-model comparison workflow: one bottom composer broadcasts the same instruction to 2, 3, 4, 6, or 8 panes while each pane keeps its own session, engine, model, mode, prompt profile, and sampling parameters.
- Adds `MultiWindowDispatchBatchSummary` so every broadcast produces batch-level observability for total panes, dispatchable panes, success/failed/skipped panes, requested/effective concurrency, `maxObservedConcurrency`, and duration.
- Adds retry failed panes behavior so partial batches can resend only failed windows while successful and skipped windows keep their current state.
- Adds cancelled multi-window batches as a governed lifecycle state so obsolete dispatches can be aborted and ignored without stale UI writeback.
- Adds manual window add and close so the multi-window surface starts empty and users explicitly choose which comparison panes are active.
- Adds manual cancel current batch control in the bottom composer so long-running multi-window dispatches can be stopped without accepting stale results.
- Adds quota-safe workspace persistence so localStorage `QuotaExceededError` falls back to same-session state instead of crashing the multi-window page.
- Carries multi-window turn options through `BirdCoderCodingSessionTurnOptions`, the core write facade, the API-backed project service, Rust server/native session payloads, codeengine provider config, and the official SDK bridge contract.
- Raises governance coverage to the current 101-check baseline by adding `codeengine-turn-options-provider` and making multi-window release writeback part of `check:multiwindow-standard`.

## Notes

- Multi-window programming now has the same first-class package and governance treatment as the established Code and Studio product surfaces.
- The provider turn-options lane deliberately standardizes only the normalized `temperature`, `topP`, and `maxTokens` fields that can be carried consistently through the current core turn API and provider bridge surfaces.
- The bottom composer now consumes the runtime batch summary instead of recomputing a parallel UI-only summary, keeping product presentation aligned with dispatch evidence.
- Manual window add and close prevents implicit session selection on entry while keeping active panes bounded by the eight-window concurrency cap.
- Failed-pane retry is handled through the shared runtime selector so retry targeting preserves first failure order and avoids duplicate pane dispatch.
- Active batch ids and AbortController ownership keep workspace switches and later batches from accepting stale high-concurrency pane events.
- Quota-safe workspace persistence keeps pane layout/model settings available in memory when browser localStorage is full, and clears the volatile fallback after a later durable write succeeds.
- The OpenCode lane remains governed by its existing official SDK bridge contract; this release does not invent an unreliable per-turn options mapping where the current SDK prompt surface does not expose a matching stable field.

## Post-release operations

- Observation goal: keep the multi-window comparison workflow, batch summary observability, turn-options propagation, and release governance baseline aligned as provider SDK mappings evolve.
- Observation window: after any edit to `packages/sdkwork-birdcoder-multiwindow`, `BirdCoderCodingSessionTurnOptions`, provider turn config records, or the official SDK bridge, rerun the verification commands above before promoting the next release note.
- Stop-ship signals: `check:multiwindow-standard` drops `multiwindow-release-writeback-contract.test.mjs`; `codeengine-turn-options-provider` disappears from governance regression; multi-window messages lose execution-profile metadata; `MultiWindowDispatchBatchSummary` stops exposing `maxObservedConcurrency`; cancelled multi-window batches stop reporting lifecycle state; manual window add and close regresses to implicit default session selection; manual cancel current batch disappears from the bottom composer; quota-safe workspace persistence stops swallowing `QuotaExceededError`; or `BirdCoderCodingSessionTurnOptions` stops reaching provider turn config.
- Release kind: `canary`
- Rollout stage: `ring-1`
- Formal or GA status: `clear`
- Machine stop-ship signals: `none`
- Rollback entry: `pnpm release:rollback:plan -- --release-tag release-2026-04-28-01 --release-assets-dir artifacts/release`
- Re-issue path: restore multi-window package/runtime/page contracts -> restore turn-options provider propagation -> rerun `pnpm.cmd run check:multiwindow-standard`, `pnpm.cmd run test:codeengine-turn-options-provider-contract`, and `pnpm.cmd lint` -> update release registry and notes together.
