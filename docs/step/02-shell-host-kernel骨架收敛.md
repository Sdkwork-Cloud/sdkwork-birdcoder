# Step 02 - Shell/Host/Kernel 骨架收敛

## 1. 目标与范围

建立 BirdCoder 统一宿主骨架，确保 `web / desktop / server` 共用同一 Kernel 入口、Host 描述和目录边界。

## 2. 执行输入

- `/docs/架构/02`、`03`、`04`、`05`
- Step `01` 基线与风险清单

## 3. 本步非目标

- 不实现 Engine Adapter 细节
- 不实现业务页面功能
- 不定义数据库迁移细则

## 4. 最小输出

- `Shell -> Host -> Kernel -> Feature` 装配链
- 宿主描述与能力注册接口
- 目录职责表
- 依赖方向检查规则

## 5. 推荐 review 产物

- 宿主拓扑图
- 目录职责表
- 依赖方向检查结果

## 6. 推荐并行车道

- 执行模式：串行
- 车道 A：入口链与宿主描述
- 车道 B：依赖方向审计

## 7. 架构能力闭环判定

- `shell / host-core / core / infrastructure` 边界稳定
- UI 与宿主不再反向侵入 Kernel

## 8. 完成后必须回写的架构文档

- `/docs/架构/02-架构标准与总体设计.md`
- `/docs/架构/03-模块规划与边界.md`
- `/docs/架构/04-技术选型与可插拔策略.md`
- `/docs/架构/05-统一Kernel与Code Engine标准.md`

## 9. 设计

- 宿主只负责描述自身能力与运行模式
- Kernel 只负责装配与统一服务出口
- 功能模块只能向 Kernel 注册，不能反向定义宿主协议

## 10. 实施落地规划

1. 收敛应用启动链与 Provider 装配顺序。
2. 冻结 Host Descriptor、运行模式、能力注入接口。
3. 清理双入口、旧命名、兼容分叉。
4. 建立高风险目录依赖检查。

## 11. 测试计划

- 宿主入口一致性检查
- 目录依赖方向检查
- 统一启动链 Smoke

## 12. 结果验证

- 三类宿主共享统一入口链
- 后续能力都可挂到稳定宿主骨架上

## 13. 检查点

- `CP02-1`：统一入口链完成
- `CP02-2`：Host 描述与注册接口冻结
- `CP02-3`：依赖方向检查通过

## 14. 风险与回滚

- 风险：边界未稳就进入功能开发，会导致后续大面积返工
- 回滚：若宿主模型冲突，回退到统一入口与目录职责表，不保留平行入口

## 15. 完成定义

- 共核宿主骨架稳定，后续 Step 不再反复改入口

## 16. 快速并行执行建议

- 并行只允许“入口链整理”和“依赖审计”
- 最终边界定义必须由 1 个总控 Owner 收口

## 17. 下一步准入条件

- 领域模型、DTO、协议可在稳定宿主骨架上冻结
