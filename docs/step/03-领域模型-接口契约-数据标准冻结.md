# Step 03 - 领域模型、接口契约与数据标准冻结

## 1. 目标与范围

冻结 BirdCoder 统一数据内核与统一协议内核，保证 `appbase`、Workbench、Terminal、Studio、Server、`coding-server` 共用同一套实体、DTO、Repository 与迁移标准。

## 2. 执行输入

- `/docs/架构/05`、`07`、`18`、`19`、`20`
- Step `02` 骨架边界

## 3. 本步非目标

- 不实现具体页面功能
- 不落地 Engine 私有协议适配
- 不做部署与发布链脚本

## 4. 最小输出

- 统一实体清单
- 统一 DTO 清单
- Repository/UoW/Provider 接口
- Canonical 资源命名
- 迁移注册主标准

## 5. 推荐 review 产物

- 实体关系图
- DTO/错误模型表
- Provider 与迁移主标准

## 6. 推荐并行车道

- 执行模式：串行
- 车道 A：实体与 DTO
- 车道 B：Repository 与迁移标准只读审计

## 7. 架构能力闭环判定

- 实体、DTO、Repository、Provider 命名一致
- `sqlite / postgresql` 后续扩展不需修改业务 API

## 8. 完成后必须回写的架构文档

- `/docs/架构/05-统一Kernel与Code Engine标准.md`
- `/docs/架构/07-数据模型-状态模型-接口契约.md`
- `/docs/架构/18-多数据库抽象-Provider-迁移标准.md`
- `/docs/架构/19-统一会话运行时-Prompt-SkillHub-AppTemplate标准.md`
- `/docs/架构/20-统一Rust-Coding-Server-API-协议标准.md`

## 9. 设计

- 统一主实体：`workspace / project / coding_session / coding_session_runtime / coding_session_turn / coding_session_message / coding_session_event / coding_session_artifact / coding_session_checkpoint`
- 统一 DTO 主源：`packages/sdkwork-birdcoder-types`
- 统一持久化边界：业务层只依赖 Repository，不直连 `localStorage`、Tauri 命令或 SQL

## 10. 实施落地规划

1. 冻结基础字段、实体主键、索引与软删语义。
2. 冻结 `coding_session_*`、`engine_registry`、`model_catalog`、`run_configuration` 等核心实体。
3. 冻结 `core / app / admin` DTO 主标准与错误模型。
4. 冻结 `runtime-data-kernel-v2` 迁移注册标准。

## 11. 测试计划

- `pnpm.cmd run check:data-kernel`
- `pnpm.cmd run typecheck`
- DTO 与 Schema 契约校验

## 12. 结果验证

- 实体、DTO、Repository、迁移命名一致
- 三端与多库共享同一逻辑语义

## 13. 检查点

- `CP03-1`：实体与索引冻结
- `CP03-2`：DTO 与错误模型冻结
- `CP03-3`：Repository/Provider/迁移标准冻结

## 14. 风险与回滚

- 风险：模型冻结不稳会导致后续 Step 大面积漂移
- 回滚：若命名或字段冲突，先修订实体与 DTO 主标准，再恢复实现

## 15. 完成定义

- BirdCoder 已拥有统一数据内核与统一 DTO 主源

## 16. 快速并行执行建议

- 并行只做实体关系整理与 DTO 草案审校
- 最终字段、命名、迁移标准必须由 1 个 Owner 收口

## 17. 下一步准入条件

- `04`、`05`、`15-18` 必须以本步冻结结果为基础推进
