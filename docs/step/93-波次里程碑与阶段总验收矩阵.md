# 93 - 波次里程碑与阶段总验收矩阵

## 1. 目标与范围

用于波次级里程碑管理和阶段总验收，确保 BirdCoder 不是只在单个 Step 上局部完成，而是在波次级达到可继续推进的整体状态。

## 2. 架构对齐

对齐 `/docs/架构/14` 与本目录 `README` 中定义的五个主波次加一个外部窗口。任何主波次未通过总验收，不允许跨波次推进。

## 3. 设计要点

- 里程碑判断的是“阶段能力是否可用”，不是“文件是否改完”。
- 阶段验收必须同时覆盖设计、实现、测试、证据、回滚准备度。
- `14-18` 是波次级架构收口项，不是可延后到发布后的附录工作。

## 4. 阶段总验收矩阵

| 波次 | 覆盖 Step | 里程碑目标 | 入口条件 | 退出条件 |
| --- | --- | --- | --- | --- |
| A | `00-03` | 冻结门禁、宿主骨架、领域模型、DTO、数据标准 | `/docs/架构/` 已冻结基线 | Shell/Host/Kernel 与领域契约稳定 |
| B | `04-05` | 冻结 Workspace/Project 语义、Engine SPI、统一会话内核 | 波次 A 通过 | 工作台上下文、Engine SPI、Session Kernel 不再漂移 |
| C | `06/07/14/15/16/18` | 并行完成两大主视图与四个架构收口专项 | 波次 B 通过 | Code/Studio 与 Appbase/多库/PromptTemplate/Engine Adapter 全部可被统一消费 |
| D | `09/17` | 完成统一 Rust `coding-server`、`core/app/admin` API、App/Admin Console | 波次 C 主能力通过 | 三端统一 API、控制台生命周期资源闭环 |
| E | `10-13` | 完成治理门禁、打包部署、测试矩阵、发布回滚 | 波次 D 通过 | 治理、交付、质量、发布全链路闭环 |
| X 外部窗口 | `08` | 在外部 Terminal 工程版本冻结后完成接入协议、启动映射、证据回写 | `09` `17` `18` 与外部版本冻结 | 外部 Terminal 集成不阻塞主线且可审计发布 |

阶段总验收最小项：

- 波次内全部 Step 已通过 `91` 与 `95`
- 波次内全部 Step 已完成 `97` 回写与 `98` 交付包归档
- 波次内关键脚本与命令可重复执行
- 必要证据均可回溯到具体能力和具体结果
- `99` 中对应解锁关系已全部满足

最快推进建议：

- 波次 A、B 只允许总控串行推进；其余车道最多准备脚手架、审计和测试草案。
- 波次 C 按 `06/07` 与 `14/15/16/18` 两组并行拆分；共享 DTO、Canonical Event、Migration、OpenAPI 统一由总控日更冻结。
- 波次 D 只在波次 C 全部通过 `91`、`95`、`98` 后启动，避免把 `09` 做成半成品底座。
- 波次 E 只在 `17` 稳定后启动正式闭环；否则治理、交付、测试会围绕错误 API 重做。
- Step `08` 不属于主波次关键路径，只能在 `09/17/18` 与外部 Terminal 版本冻结后开启。

## 5. 测试与结果验证

- 每个波次结束时必须进行一次阶段级验收或综合演练。
- 波次总验收结论必须可复核，不允许口头通过。
- 若某个 Step 已完成但阶段目标未达成，仍判定该波次未通过。

## 6. 检查点

- `CP93-1`：五个波次的入口、退出条件全部明确。
- `CP93-2`：波次 C 与外部窗口的职责已明确拆开，Step `08` 不再混入主交付波次。
- `CP93-3`：阶段验收能直接决定是否进入下一波次。

## 7. 执行方式

| 项 | 说明 |
| --- | --- |
| 使用时机 | 每个波次启动前和结束后 |
| 是否可并行 | 波次内可并行准备证据，总验收结论必须串行收口 |
| 输出要求 | 里程碑结论、阶段证据、准入/退回判断 |
