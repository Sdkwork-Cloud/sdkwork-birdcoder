# 96 - 最优并行执行周计划与资源编排

## 1. 目标与范围

给出 BirdCoder Step 体系的最快安全执行节奏，明确每周波次、车道人数、集成窗口和阻塞升级时机，在不破坏统一 Kernel、统一数据标准、统一 API 的前提下，把并行效率拉满。

## 2. 架构对齐

对齐本目录 `README`、`94`、`99` 与 `/docs/架构/14`。任何偏离主链 `00-05 -> 并行波次 -> 09 -> 17 -> 10/11/12 -> 13` 的安排，都不能视为最优路径。

## 3. 推荐资源编排

| 车道 | 人数建议 | 负责 Step | 主写目录 |
| --- | --- | --- | --- |
| A 总控 | 1 | `00-05` `09` `17` `13` | `packages/sdkwork-birdcoder-core` `packages/sdkwork-birdcoder-server` `.github` `docs/release` |
| B Code | 1-2 | `06` | `packages/sdkwork-birdcoder-code` `packages/sdkwork-birdcoder-ui` |
| C Studio | 1-2 | `07` | `packages/sdkwork-birdcoder-studio` `packages/sdkwork-birdcoder-host-studio` `packages/sdkwork-birdcoder-templates` |
| D External 集成 | 0-1 | `08` | `packages/sdkwork-birdcoder-terminal` `packages/sdkwork-birdcoder-desktop` `packages/sdkwork-birdcoder-shell` |
| E 架构补强 | 2-4 | `14` `15` `16` `18` | `packages/sdkwork-birdcoder-appbase` `packages/sdkwork-birdcoder-types` `packages/sdkwork-birdcoder-infrastructure` `docs/prompts` |
| F 治理交付 | 2-3 | `10` `11` `12` | `deploy` `scripts` `artifacts` `.github/workflows` |

最小高效配置：`6` 人。推荐配置：`8-10` 人。若资源不足，优先保障 `A + B + C + E + F`；`D` 只在外部窗口开启时补位。

## 4. 最快周计划

| 周次 | 主目标 | 车道动作 | 出口条件 |
| --- | --- | --- | --- |
| W1 | 完成 `00-03` | A 串行冻结；其余车道只做审计与脚手架 | 门禁、骨架、模型、DTO 冻结 |
| W2 | 完成 `04-05` | A 串行；E 预热 `14/15/16/18` 草案 | 工作台语义与 Engine SPI 冻结 |
| W3-W5 | 并行完成 `06/07/14/15/16/18` | B/C/E 主实现并行；D 维护外部 Terminal 版本矩阵与接入草案；A 守共享契约 | Code/Studio 主视图与四个专项均达准入 |
| W6 | 完成 `09` | A 串行做 `coding-server` 与双模收口 | 三端统一访问与 DTO 稳定 |
| W7 | 完成 `17` | A 主导 `core/app/admin` API 与 Console 收口 | 生命周期资源进入统一 API |
| W8-W9 | 并行完成 `10/11/12` | F 拆治理/交付/质量三线并行 | 门禁、打包部署、测试矩阵闭环 |
| W10 | 完成 `13` | A 主导发布演练；全员辅助核证据 | 发布、灰度、回滚闭环 |

加速说明：

- 若团队达到 `10+` 人，W3-W5 可压缩为 W3-W4。
- 若 E 车道只剩一个人，优先顺序为 `14 -> 15 -> 16 -> 18`。
- 固定串行闸门：`00-05`、`09`、`17`、`13`。
- 最大收益并行窗口：`06/07/14/15/16/18`。
- 任一车道触发 `B2/B3`，当日集成窗口直接切回 A 总控收口。

外部窗口说明：

- Step `08` 只在 `09`、`17`、`18` 稳定且外部 Terminal 工程版本冻结后开启。
- 外部窗口不进入本仓主交付周计划，不得反向阻塞 `Code / Studio` 主线。

## 5. 每日节奏

- 上午：各车道独立推进，不改共享契约。
- 中午：15 分钟同步阻塞、契约变更、目录冲突。
- 下午：继续本车道实现。
- 收工前：统一集成窗口，只处理共享契约与主链验证。

## 6. 检查点

- `CP96-1`：资源分配与车道角色冻结。
- `CP96-2`：每周出口条件可量化。
- `CP96-3`：每日集成窗口固定，阻塞升级不拖延。

## 7. 执行方式

| 项 | 说明 |
| --- | --- |
| 使用时机 | 启动全局实施前、波次切换前 |
| 是否可并行 | 计划可并行评审，发布版本只允许一个主节奏 |
| 输出要求 | 周计划、车道人数、每日节奏、出口条件 |
