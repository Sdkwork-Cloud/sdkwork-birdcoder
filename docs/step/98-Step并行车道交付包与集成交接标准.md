# 98 - Step 并行车道交付包与集成交接标准

## 1. 目标与范围

定义多车道并行时每条车道必须交付什么、如何交接、如何进入集成窗口，避免“代码写完了，但没人能接、没人敢合”。

## 2. 架构对齐

对齐 `94`、`96` 与 `/docs/架构/10`。未满足交付包标准的车道成果，不得进入主集成。

## 3. 标准交付包

每条车道最小交付包必须包含：

- 变更说明：做了什么、没做什么、影响什么
- 写入目录清单：精确到包或目录
- 契约影响：新增、删除、变更的接口、字段、命名
- 测试结果：已跑命令、通过结果、未覆盖项
- 风险说明：已知风险、回滚点、阻塞项
- 证据索引：日志、报告、产物、截图路径
- 解锁说明：对应 `99` 的完成信号与下一解锁项

## 4. 车道交接标准

| 车道 | 交接重点 | 集成前必须确认 |
| --- | --- | --- |
| A 总控 | 共享 DTO、Canonical Event、Migration、OpenAPI、最终合并顺序 | 共享契约未漂移 |
| B Code | 编辑器、文件系统、Code 工作台入口 | 不破坏 Studio 与外部 Terminal 启动映射 |
| C Studio | Preview、Simulator、Build、Template 挂接 | 预览与真实构建一致 |
| D External 集成 | 外部 Terminal contract、启动桥接、版本矩阵、证据接口 | 外部版本、请求结构、审计回写未漂移 |
| E 架构补强 | Appbase、多库、PromptTemplate、Engine Adapter | 不破坏统一实体、Repository、Tool/Event 标准 |
| F 治理交付 | 脚本、部署、门禁、测试、发布证据 | 产物矩阵、Smoke、CI 标准一致 |

## 5. 集成顺序

- 共享契约
- Appbase / Provider / PromptTemplate / Engine Adapter
- Code / Studio 主视图
- 外部 Terminal 集成窗口（如开启）
- `coding-server` / `core/app/admin` / Console
- 治理、部署、测试矩阵
- 发布与回滚

规则：

- 每轮并行只允许 1 个固定集成窗口。
- 若任一车道未提供完整交付包，整轮不集成。
- 集成窗口只解决共享边界问题，不重新讨论需求。

最快交接建议：

- E 车道先提交共享契约包，再提交各自实现包；不要把契约和实现混成一次大交付。
- B/C 只在拿到当天共享契约冻结版后提交主视图交付包；D 只在外部版本冻结后提交集成交付包。
- A 总控只在 `09`、`17`、`13` 三个主链收口点接入跨车道成果，避免持续交叉合并。
- F 车道对 `10/11/12` 只接收已通过 `09/17` 主链验证的产物，不对未冻结 API 反向适配。

## 6. 检查点

- `CP98-1`：每个车道均已提供最小交付包。
- `CP98-2`：共享契约变更均已被总控复核。
- `CP98-3`：集成窗口可在一次主链验证中完成收口。

## 7. 执行方式

| 项 | 说明 |
| --- | --- |
| 使用时机 | 每轮并行结束前、进入集成窗口前 |
| 是否可并行 | 各车道可并行准备交付包，集成必须串行执行 |
| 输出要求 | 交付包、风险说明、测试结果、证据索引 |
