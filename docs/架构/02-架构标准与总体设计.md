# 02-架构标准与总体设计

## 1. 总体原则

- Package First：先冻结包边界，再进入实现。
- Kernel First：先统一会话、工具、证据、治理模型，再接入具体引擎。
- API First：`web / desktop / server` 只消费统一 `coding-server` 协议。
- Release First：开发期即对齐打包、Smoke、Finalize、Release Notes。
- External Integration First：非核心能力优先通过稳定接口集成外部工程，不在本仓重复实现。

## 2. 当前产品边界

- BirdCoder 是新型 AI IDE，当前产品核心聚焦 `Code` 与 `Studio`。
- `Terminal` 不再作为本仓主线实现目标；真实终端、PTY、CLI 会话、命令执行与交互体验由独立外部工程提供。
- BirdCoder 对 Terminal 只负责四类能力：
  - 外部工程接入协议
  - 启动与上下文映射
  - 审计 / 发布回写接口
  - 宿主侧集成入口
- 底层统一引擎继续覆盖 Claude Code、Codex、Gemini、OpenCode，并保持后续可扩展。
- `auth / user / vip` 的标准能力统一来自 `sdkwork-appbase`，BirdCoder 只保留 `sdkwork-birdcoder-auth`、`sdkwork-birdcoder-user` 这两个薄适配包，不在本仓重建平行用户中心。

## 3. 技术分层

```text
Host Layer
  web / desktop / server

Shell Layer
  bootstrap / routing / theme / appbase injection / host policies

Kernel Layer
  coding-session orchestration / policy / audit / artifact / execution orchestration

Service Layer
  coding-server / core-api / app-api / admin-api / openapi / sdk generation

Domain Layer
  code / studio / prompt / skills / templates / settings / release / external integrations

Adapter Layer
  engine / filesystem / git / build / preview / simulator / test / deploy / mcp / storage / external-terminal bridge

Delivery Layer
  distribution / release assets / docker / kubernetes / ci / smoke / github release
```

## 4. 业务分层

1. `Workspace Layer`：工作空间、偏好、权限作用域。
2. `Project Layer`：项目、工程配置、引擎绑定、模板实例。
3. `Coding Session Layer`：统一编码会话与消息投影。
4. `Code Workspace Layer`：文件、Diff、Git、索引、上下文装配。
5. `Studio Execution Layer`：Build、Preview、Simulator、Test、Evidence。
6. `Prompt/Skill/Template Layer`：Prompt、SkillHub、App Template、项目模板注入。
7. `Integration Layer`：外部 Terminal、外部工具链、外部交付系统接入。
8. `Release Layer`：发布、部署、回滚、治理与证据归档。

说明：

- `Execution Layer` 的本仓主线只包含 `Build / Preview / Simulator / Test / Release`。
- `Terminal` 在本架构中属于 `Integration Layer`，不是本仓内建运行时。

## 5. 统一服务协议标准

- 所有宿主统一通过 Rust `coding-server` 演进 `core / app / admin` 协议面。
- 页面层禁止直接依赖引擎私有 CLI / SDK / HTTP 协议。
- 外部 Terminal 也必须通过标准接入层进入：
  - 启动请求由 BirdCoder 标准 DTO 发出
  - 会话 / 输出 / 命令历史 authority 留在外部工程
  - BirdCoder 只消费状态摘要、审计摘要、证据引用
- OpenAPI 只作为文档和 SDK 载体；真正冻结的是 `types -> OpenAPI -> handler/client` 的统一契约链。

## 6. 包级职责映射

- `sdkwork-birdcoder-shell`：统一壳层启动、注入、导航。
- `sdkwork-birdcoder-core`、`sdkwork-birdcoder-commons`：Kernel 契约、上下文、公共运行时。
- `sdkwork-birdcoder-codeengine`：统一多引擎目录、运行时选择、宿主支持与会话桥接收口。
- `sdkwork-birdcoder-chat`、`sdkwork-birdcoder-chat-*`：底层 provider bridge 与官方 SDK 适配实现。
- `sdkwork-birdcoder-code`：代码主工作面。
- `sdkwork-birdcoder-studio`：Studio 主工作面。
- `sdkwork-birdcoder-commons/src/terminal/*`、`sdkwork-birdcoder-shell`、`sdkwork-birdcoder-desktop`：共同承担外部 Terminal 集成入口、状态桥接、协议投影；不承担终端实现 authority。
- `sdkwork-birdcoder-skills`、`sdkwork-birdcoder-templates`：技能与模板体系。
- `sdkwork-birdcoder-auth`、`sdkwork-birdcoder-user`：对 `sdkwork-appbase` 身份、用户、会员能力的薄适配层，只承载 BirdCoder 的路由、服务注入、运行时绑定与领域映射。
- `sdkwork-birdcoder-server`：`coding-server` 与统一 API 宿主。
- `sdkwork-birdcoder-desktop`、`sdkwork-birdcoder-web`：统一 API 消费宿主。
- `sdkwork-birdcoder-distribution`：多宿主交付清单。

## 7. 当前聚焦与非目标

- 当前主线：
  - `Code`
  - `Studio`
  - `coding-server`
  - 统一 Kernel / DTO / Provider / OpenAPI / Release
- 当前非目标：
  - 本仓内实现 PTY
  - 本仓内实现 CLI Registry
  - 本仓内实现 Terminal Session / Execution authority
  - 本仓内持续深挖 Terminal UI 交互
- Terminal 后续只通过外部独立工程版本化集成。

## 8. 对齐标准

- `Code / Studio` 是产品核心主线，优先级高于 Terminal 集成。
- `Terminal` 只能通过外部工程 manifest / SDK / adapter / launch contract 集成。
- `web / desktop / server` 只消费统一 `coding-server` 与外部集成协议，不复制平行接口。
- `auth / user / vip` 只走 `appbase`。
- 本仓不得再为 Terminal 扩张本地 authority 数据模型。

## 9. 评估标准

- 主线聚焦度：资源是否优先投入 `Code / Studio / coding-server`。
- 架构清晰度：Terminal 是否被正确降级为外部集成边界。
- 契约稳定性：外部 Terminal 接入是否只经由统一 DTO / adapter。
- 宿主一致性：`web / desktop / server` 是否消费同一套接口与集成协议。
- 商业化完整性：即使 Terminal 来自外部工程，发布、审计、证据链仍完整可追溯。
