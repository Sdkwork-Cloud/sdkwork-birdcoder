# 03-模块规划与边界

## 1. 当前包结构映射

### 1.1 基础层

- `sdkwork-birdcoder-core`：初始化、事件总线、基础运行时设施。
- `sdkwork-birdcoder-types`：共享类型、DTO、协议契约与数据定义。
- `sdkwork-birdcoder-commons`：通用 Hook、上下文、公共运行时封装。
- `sdkwork-birdcoder-ui`：通用 UI 组件。
- `sdkwork-birdcoder-i18n`：国际化资源。
- `sdkwork-birdcoder-infrastructure`：平台服务接口与默认实现。

### 1.2 宿主与壳层

- `sdkwork-birdcoder-shell`：统一壳层与应用启动。
- `sdkwork-birdcoder-host-core`：宿主描述与运行模式标准。
- `sdkwork-birdcoder-host-studio`：Studio 宿主描述与预览/模拟相关适配。
- `sdkwork-birdcoder-web`、`sdkwork-birdcoder-desktop`、`sdkwork-birdcoder-server`：具体交付宿主。

### 1.3 业务能力层

- `sdkwork-birdcoder-code`：Code 主工作面。
- `sdkwork-birdcoder-studio`：Studio 主工作面。
- `sdkwork-birdcoder-terminal`：外部 Terminal 集成入口与桥接层，不是终端实现层。
- `sdkwork-birdcoder-settings`、`sdkwork-birdcoder-skills`、`sdkwork-birdcoder-templates`：辅助业务能力。
- `sdkwork-birdcoder-appbase`：统一承载 `auth / user / vip`。

### 1.4 协议与引擎层

- `sdkwork-birdcoder-server`：统一 `coding-server` 包，包含 TypeScript facade 与 Rust host。
- `sdkwork-birdcoder-chat`：引擎公共协议层。
- `sdkwork-birdcoder-chat-claude`、`codex`、`gemini`、`opencode`：引擎适配层。

### 1.5 分发与交付层

- `sdkwork-birdcoder-distribution`：发行版清单。
- `deploy/docker`、`deploy/kubernetes`：部署模板。
- `scripts/release/*`：打包、Smoke、Finalize、Release Notes。

## 2. 功能模块映射

- `code`：代码编辑、文件树、Diff、统一编码会话入口。
- `studio`：Build、Preview、Simulator、Test、Evidence、AI 工作台。
- `terminal`：外部工程接入入口，只提供启动、状态、审计、证据桥接。
- `settings`：通用设置、环境、Git、MCP、外部集成配置。
- `skills`：技能市场与技能绑定入口。
- `templates`：项目模板与应用模板入口。
- `coding-server`：统一承载项目生命周期 API。

## 3. 当前权威事实

- `sdkwork-birdcoder-types` 是当前可执行的协议 truth source。
- `sdkwork-birdcoder-server` 负责统一 `core / app / admin` API 面，不再为宿主复制协议。
- `sdkwork-birdcoder-terminal` 不是终端 authority；真实 Terminal Runtime truth 在外部独立工程。
- BirdCoder 本仓只保留：
  - 外部 Terminal 启动 contract
  - 上下文映射
  - 状态摘要与证据引用
  - 发布 / 审计回写

## 4. 依赖边界

- 业务模块不得直接依赖具体引擎私有协议。
- 页面层不得直接依赖 PTY、原生 CLI、终端会话存储或外部工程私有 SDK。
- `sdkwork-birdcoder-terminal` 只能依赖：
  - `@sdkwork/birdcoder-types`
  - `@sdkwork/birdcoder-commons`
  - `@sdkwork/birdcoder-infrastructure`
  - 宿主桥接接口
- 本仓任何包都不得再扩张 Terminal Session / Execution 的 authority 表、Repository、SQLite 迁移或 UI 内部协议。
- 部署脚本不感知页面细节，只消费标准产物与 manifest。

## 5. 组件化规则

- `Code` 与 `Studio` 是本仓一等工作面，允许独立拆分、独立深化、独立优化。
- `CodePage` 只保留项目/会话/文件/运行状态编排；搜索浮层、编辑区、Diff、侧边聊天、运行弹窗、外部 terminal 集成都必须下沉到独立子组件。
- 本地目录挂载统一走 `LocalFolderMountSource -> IFileSystemService.mountFolder(projectId, folderInfo) -> useFileSystem().mountFolder(projectId, folderInfo)`；页面层不得直接访问 `fileSystemService` 或使用 `as any` 绕过边界。
- Terminal 相关组件在本仓只允许是 facade / adapter / integration panel。
- 任何 Terminal 相关 UI 都不得在本仓内重新发明会话模型、输出模型、恢复模型。
- 引擎适配器必须实现统一 SPI，不允许把引擎分支散落到页面层。
- `chat / editors / run-config` 等重型 UI 子路径可以由子组件直接消费，但不得重新回流到页面壳层，避免主工作面重新膨胀。

## 6. 扩展规则

- 新增 Code Engine：新增 adapter 与能力声明，不改主工作流。
- 新增宿主：新增 host adapter，不改 Kernel。
- 新增 Studio 能力：新增 preview/build/simulator/test adapter，不改会话主语义。
- 新增 Terminal 能力：优先在外部工程实现，本仓只扩展集成协议、状态桥接与证据接口。

## 7. 评估标准

- 边界是否清晰：Code / Studio / External Terminal 是否职责单一。
- 主线是否聚焦：本仓资源是否优先投入 Code / Studio / coding-server。
- 集成是否克制：Terminal 是否只通过外部 contract 接入。
- 变更是否低耦合：新增 Terminal 能力是否无需改动 Code / Studio 主链路。

## 8. Step 06 command-boundary addendum

- `CodePage` may keep page state and emit intents, but it must not own `globalEventBus.on/off` workbench subscriptions.
- `useCodeWorkbenchCommands()` is the Code workbench command boundary for `toggleDiffPanel`, `startDebugging`, `runWithoutDebugging`, `addRunConfiguration`, `runTask`, `findInFiles`, `openQuickOpen`, session navigation, and related shell-intent commands.
- The dedicated hook keeps editor/diff/run-entry orchestration isolated from the page shell in the same way `CodeWorkspaceOverlays`, `CodeEditorWorkspacePanel`, `CodeEditorSurface`, `CodePageDialogs`, and `CodeTerminalIntegrationPanel` isolate their UI/runtime lanes.
- `scripts/code-workbench-command-boundary-contract.test.mjs` is the architecture guard; `pnpm.cmd run check:code-workbench-command-boundary` must stay inside both `lint` and `check:release-flow`.
- `useCodeRunEntryActions()` is the Code run-entry boundary for run-configuration persistence, launch resolution, blocked-launch messaging, task execution dispatch, and debug launch fallback.
- `CodePage` must not call `useProjectRunConfigurations()`, `resolveRunConfigurationTerminalLaunch()`, or `buildTerminalProfileBlockedMessage()` directly after the run-entry cut.
- `scripts/code-run-entry-boundary-contract.test.mjs` is the architecture guard for that second slice; `pnpm.cmd run check:code-run-entry-boundary` must stay inside both `lint` and `check:release-flow`.
- `CodeEditorSurface` is the only valid Code-side boundary for `CodeEditor`, `DiffEditor`, diff accept or reject controls, and editor empty-state CTA or copy.
- `CodeEditorWorkspacePanel` must not import `CodeEditor` or `DiffEditor` directly, and it must not own diff header buttons or editor empty-state icon or CTA details after the surface split.
- `scripts/code-editor-surface-boundary-contract.test.mjs` is the architecture guard for that third slice; `pnpm.cmd run check:code-editor-surface-boundary` must stay inside both `lint` and `check:release-flow`.
