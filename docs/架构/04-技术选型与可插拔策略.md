# 04-技术选型与可插拔策略

## 1. 当前技术基线

| 领域 | 选型 | 说明 |
| --- | --- | --- |
| Workspace | `pnpm workspace` | 统一多包管理与脚本入口 |
| 语言 | `TypeScript + Rust` | 前端、共享契约与服务宿主统一建模 |
| UI | `React 19` | 组件化、状态驱动、跨宿主复用 |
| 构建 | `Vite` | 快速开发构建与多包集成 |
| 桌面 | `Tauri 2 + Rust` | 原生能力、安装包产物、较轻运行时 |
| 服务端 | `Rust coding-server + axum` | 统一 `core/app/admin` API 与事件流中枢 |
| 引擎接入 | `Engine SPI + Adapter` | 屏蔽 CLI、SDK、HTTP、SSE、JSON-RPC 差异 |
| 权威存储 | `SQLite + PostgreSQL` | 同一逻辑 schema，按宿主选择 Provider |
| 浏览器缓存 | `localStorage / IndexedDB` | 仅承担 UI cache，不进入权威持久化链路 |
| 文档 | `VitePress` | 文档即站点、易发布 |
| 部署 | `Docker / Helm` | Desktop、Server、Docker、K8s 统一交付 |

## 2. 选型原则

- 优先统一而非堆技术栈。
- 优先可替换而非一次性耦死。
- 优先工程化闭环而非单点最优。
- 优先跨平台打包一致性而非局部方便。
- 优先协议与 SDK 稳定性，而非页面直连或临时脚本接入。

## 3. Code Engine 协议与 SDK 策略

| 引擎 | 首选接入 | 原生协议事实 | BirdCoder 接入策略 |
| --- | --- | --- | --- |
| Codex | `@openai/codex-sdk` + app-server SDK | CLI JSONL；Python 侧有 JSON-RPC v2；`thread -> turn -> item/event` | 统一映射为 `coding_session_turn/event`，同时保留 CLI 本地宿主与 server 宿主双路径 |
| Gemini | `gemini-cli-core` 衍生 SDK | `sessionId`、事件流、Tool Registry、Skill 装载、动态 instructions | 复用原生 session/tool/skill 能力，不在 UI 层重造工具协议 |
| OpenCode | `@opencode-ai/sdk` | OpenAPI/HTTP + SSE；`session/message/part`；自定义工具、结构化输出 | 直接消费类型安全 SDK，把 `diff/todo/pty/question` 归一为事件与 Artifact |
| Claude Code | remote-control server + SDK adapter | `session/code-session`、bridge、history、SSE、SDK message adapter | 通过远程桥接与消息适配接入，统一投影 `stream/progress/result` |

## 4. 可插拔能力域

- Code Engine：Claude Code、Codex、Gemini、OpenCode。
- Tool Provider：文件、终端、Git、搜索、构建、测试、预览。
- Build Provider：Web、Desktop、Server、Container、K8s。
- Preview Provider：浏览器预览、桌面预览、设备模拟器、接口仿真器。
- Auth Provider：本地令牌、团队登录、企业代理。
- Storage Provider：SQLite、PostgreSQL、BlobStore。
- Prompt / SkillHub / App Template Provider：规则包、技能包、模板包、行业工作流。
- Distribution Provider：`global`、`cn` 及未来区域化发行版。

## 5. 替换门槛

- 新选型必须通过统一契约接入。
- 必须提供替换边界、迁移路径、数据清理方案和统一契约说明。
- 不允许仅因局部实现便利而破坏多宿主与多引擎一致性。
- 新引擎接入前必须确认：会话连续性、工具保真度、结构化输出、审批语义、Artifact 提取、SDK 可维护性。

## 6. 评估标准

- 技术是否降低了模块耦合。
- 技术是否改善了开发、构建、发布效率。
- 技术是否支持多平台、多架构、多部署模式。
- 技术是否具备长期维护成本可控性。
- 是否能被 `coding-server` 统一投影为同一套 DTO、错误模型和 SSE 事件。
