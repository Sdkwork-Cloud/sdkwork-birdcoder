# 06-编译环境-预览-模拟器-测试体系

## 1. 目标

BirdCoder 的 Studio 侧必须把 Preview、Build、Simulator、Test 统一到同一执行内核，统一复用运行配置、终端启动守卫、证据模型和 release 归档标准，避免四套平行实现。

## 2. 统一执行分层

- Profile Layer
  - Preview 使用 `PreviewSession`
  - Build 使用 `BuildProfileRegistry`
  - Simulator 使用 `HostStudioSimulatorSession`
- Launch Layer
  - 全部通过共享 `resolveRunConfigurationTerminalLaunch(...)` 生成 terminal request。
  - 缺失 CLI、PATH 未就绪、profile 不可启动时使用同一阻断与恢复动作。
- Evidence Layer
  - Preview、Build、Simulator 均输出标准 execution evidence。
  - 存储层统一按项目隔离、按时间排序、按 evidenceKey 去重。
- UI Layer
  - StudioPage 已接入 Preview 与 Simulator 两类真实入口。
  - `StudioPage` 只保留编排职责；project/chat 侧栏、工作区搜索浮层、外部 terminal 集成分别下沉到 `StudioChatSidebar`、`StudioWorkspaceOverlays`、`StudioTerminalIntegrationPanel`。
  - `CodePage` 同样采用编排壳标准；工作区搜索浮层、编辑器/Diff/侧边聊天工作区、运行弹窗、外部 terminal 集成分别下沉到 `CodeWorkspaceOverlays`、`CodeEditorWorkspacePanel`、`CodePageDialogs`、`CodeTerminalIntegrationPanel`。
  - Simulator 必须继续通过独立 `StudioSimulatorPanel` 接入，不允许回流为 `StudioPage` 内联大块 UI。
  - Build 通过 Run Task 入口进入统一执行链路。
  - Test 通过 Run Task 入口进入统一执行与本地证据存储链路。
  - Studio 右侧已提供 unified evidence viewer，统一展示 Preview / Build / Simulator / Test 最新执行证据。
  - viewer 已支持 lane filter，优先解决多 lane 混排下的可读性问题。
  - viewer 已支持 Replay，并直接复用既有 terminal request 总线。
  - viewer 已支持 Export，可下载当前过滤后的证据 JSON 包。
- Release Layer
  - Preview 已进入 `release-manifest.json.previewEvidence` 与 finalized smoke。
  - Build 已可选进入 `release-manifest.json.buildEvidence`，raw archive 为 `studio/build/studio-build-evidence.json`。
  - Simulator 已可选进入 `release-manifest.json.simulatorEvidence`，raw archive 为 `studio/simulator/studio-simulator-evidence.json`。
  - Test 已可选进入 `release-manifest.json.testEvidence`，raw archive 为 `studio/test/studio-test-evidence.json`。
  - finalized smoke 会对所有已附带的 Studio evidence summary 做 raw archive 一致性校验。
  - 当前四条 lane 的统一性以共享 launch contract、evidence contract 与 release archive contract 为主；服务端侧统一编排仍以后续 `coding-server` 落地为准。

## 3. 当前标准

### 3.1 Build

- 目标解析：`resolveStudioBuildProfile(...)`
- 执行解析：`resolveStudioBuildExecutionLaunch(...)`
- 证据存储：`build-evidence.{projectId|global}.v1`

### 3.2 Simulator

- 宿主目标：`resolveHostStudioSimulatorSession(...)`
- 执行解析：`resolveStudioSimulatorExecutionLaunch(...)`
- 证据存储：`simulator-evidence.{projectId|global}.v1`
- UI 接入：`StudioPage` simulator mode

## 4. 评估标准

| 维度 | 标准 |
| --- | --- |
| 一致性 | Preview / Build / Simulator / Test 必须共用运行配置、守卫和证据结构 |
| 低耦合 | Release 层只能读取归一化摘要或归档文件，不允许反向耦合 UI 本地结构 |
| 可回放 | 任一执行请求都能用标准 terminal request 和 evidence 进行诊断与复现 |
| 可门禁 | 契约测试必须进入 `check:release-flow`、结构检查和架构检查；`StudioPage` 与 `CodePage` 组件化边界分别由 `studio-page-componentization-contract.test.mjs`、`code-page-componentization-contract.test.mjs` 冻结 |

## 5. 当前差距

- Test lane 已完成 Studio 执行 envelope、进入 unified viewer，并进入 release 摘要与 finalized smoke。
- viewer 已补齐面向 diagnostics / release triage 的标准化 export 基础能力，后续继续增强更细粒度闭环。

## 6. 演进顺序

1. 在统一 export 契约上持续增强 Test evidence 与 release / diagnostics / triage 的协同能力。
2. 在导出结果中固化 laneFilter、summary、project/profile scope 与 latestLaunchedAt。
3. 在统一 evidence contract 上补齐更细粒度诊断能力。
