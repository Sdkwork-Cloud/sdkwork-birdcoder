# 07-数据模型-状态模型-接口契约

## 1. 目标

BirdCoder 的数据标准采用“领域优先、存储后置、运行时分层”模式。业务层只依赖统一领域模型、仓储契约和事务边界，不直接依赖 `sqlite`、`postgresql`、`localStorage` 或页面私有结构。

## 2. 建模原则

- `auth / user / vip` 主模型统一来自 `sdkwork-appbase` 的 canonical 定义，BirdCoder 仅保留业务扩展字段、运行时绑定和投影映射。
- 统一主骨架采用 `workspace -> project -> coding_session`，不再把旧 `thread / message` 作为顶层业务容器。
- 统一引擎主键固定为 `codex / claude-code / gemini / opencode`；别名只允许进入归一化层，不进入权威存储。
- 引擎原生 ID、流、恢复点只落在 `coding_session_runtime`，不直接进入页面模型。
- UI 读标准投影，审计和排障读原始事件，证据与大对象走 Artifact/Blob 分层。
- 对外契约使用 `camelCase`，库表字段使用 `snake_case`。
- API DTO 与数据实体分层定义，但命名和状态语义必须一致。

| 领域 | 核心实体 | 说明 |

## 4. 主关系标准

```text
workspace
└─ project
   ├─ coding_session
   │  ├─ coding_session_runtime*
   │  ├─ coding_session_turn*
   │  ├─ coding_session_message*
   │  ├─ coding_session_event*
   │  ├─ coding_session_artifact*
   │  └─ coding_session_checkpoint*
   ├─ project_document*
   ├─ skill_binding*
   ├─ app_template_instantiation*
   ├─ run_configuration*
   ├─ deployment_record*
   └─ terminal/build/preview/simulator/test execution*
```

规则：

- 一个 `workspace` 下可有多个 `project` 与 `team`。
- 一个 `project` 下可有多个 `coding_session`、`project_document`、`deployment_target`、`deployment_record`。
- 一个 `coding_session` 可对应多个 `coding_session_runtime`，用于引擎切换、恢复、重连或宿主迁移。
- 大文件、补丁包、日志归档优先由 `coding_session_artifact` + Blob/FileStore 承载。

## 5. 状态模型

- `coding_session`：`draft -> active -> paused -> completed | archived`
- `coding_session_runtime`：`initializing -> ready -> streaming | awaiting_tool | awaiting_approval -> completed | failed | terminated`
- `coding_session_turn`：`queued -> running -> completed | failed | cancelled`
- `coding_session_artifact`：`draft -> sealed -> archived`
- `project_document`：`draft -> active -> archived`
- `deployment_record`：`planned -> running -> succeeded | failed | rolled_back`
- `terminal_session`：`idle -> running -> succeeded | failed | cancelled -> closed`
- `build/preview/simulator/test`：`ready -> queued -> launching -> running -> succeeded | failed | cancelled -> archived`

## 6. 统一接口契约

- `Repository<TAggregate, TId>`：聚合读写边界。
- `QueryRepository<TView>`：列表、分页、过滤、排序、投影查询。
- `UnitOfWork`：`begin / commit / rollback / withinTransaction`。
- `StorageProvider`：连接、事务、迁移、健康检查。
- `SchemaMigrator`：逻辑迁移清单与方言脚本执行。
- `BlobStore`：大对象、补丁归档、证据包、会话导出。
- `Dialect`：类型映射、占位符、分页、JSON、时间函数适配。

页面、Hooks、Tauri command、Server handler 只允许依赖这些标准接口，不允许直接写 SQL 或直连 `localStorage`。

## 7. 统一 API DTO 规则

- 所有 `coding-server` DTO 统一定义在 `packages/sdkwork-birdcoder-types`，由 Web、Desktop、Server 共用。
- `core API` DTO 面向 Kernel、引擎、会话、事件、Artifact。
- `app API` DTO 面向 workspace、project、document、template、prompt、skill、test、deployment。
- `admin API` DTO 面向 team、member、policy、audit、release、deployment governance。
- OpenAPI 生成物、Rust handler、前端 SDK、桌面回环客户端必须从同一套 DTO 推导，不允许手写平行结构。

## 8. 逻辑类型标准

| 逻辑类型 | SQLite | PostgreSQL | 规则 |
| --- | --- | --- | --- |
| `id` | `TEXT` | `UUID`/`TEXT` | 对外统一字符串 ID |
| `json` | `TEXT` | `JSONB` | 结构先定义 schema，再决定是否落 JSON |
| `enum` | `TEXT` | `TEXT`/`ENUM` | 优先字符串枚举，避免方言锁定 |
| `timestamp` | `INTEGER/TEXT` | `TIMESTAMPTZ` | 对外统一 ISO 8601 |
| `blob-ref` | `TEXT` | `TEXT` | 大对象只保存引用，不把超大内容直接塞入主表 |

## 9. 切换原则

- 领域主模型固定为 `workspace / project / coding_session / coding_session_runtime / coding_session_turn / coding_session_message / coding_session_event / coding_session_artifact / coding_session_checkpoint`。
- 前端共享视图类型当前以 `BirdCoderProject / BirdCoderCodingSession / BirdCoderChatMessage` 呈现，但它们只是 UI projection，不是新的领域主命名。
- 存储主模型只允许 `coding_session_*`、`runtime_*`、`prompt_*`、`skill_*`、`template_*` 等标准实体，不再保留旧 `thread / message / conversation` 聚合。
- 所有 schema 变更先定义逻辑迁移 ID，再分别提供 `sqlite` 与 `postgresql` 方言脚本，并同步更新 API DTO 版本说明。
- 新增字段必须同时给出默认值、迁移脚本、数据校验和失败补偿方案。

## 10. 评估标准

- 完整性：实体、状态、仓储、索引、迁移、保留策略成套定义。
- 中立性：业务层不感知具体数据库驱动、方言和宿主。
- 可扩展性：新增引擎、Provider、模板系统时不重写主骨架。
- 可审计性：任意会话、Artifact、执行证据都能追溯到用户、项目、引擎、时间和版本。
- 可恢复性：会话中断、审批挂起、执行失败后可从 `checkpoint/runtime` 恢复。
- 协议一致性：DTO、OpenAPI、Rust handler、前端 SDK 的字段和错误语义一致。
