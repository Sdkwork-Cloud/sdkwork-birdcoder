# 08-性能-安全-可观测性标准

## 1. 性能预算与当前门禁

- 已工程化门禁：Web 包体预算固定为入口 JS 小于 500 KiB、任一 JS 产物小于 700 KiB，由 `node scripts/web-bundle-budget.test.mjs` 自动校验。
- 已冻结目标预算：Desktop 冷启动小于 5 秒、首次 AI 响应首 Token 小于 2 秒、预览增量刷新小于 1.5 秒。
- 上述预算统一定义在 `packages/sdkwork-birdcoder-types/src/governance.ts`，任何门禁脚本都必须消费共享合同。
- 除 Web 包体外，其余预算当前仍以标准和证据约束为主，待统一接入结构化指标与 release evidence 后再升级为硬门禁。

## 2. 当前风险基线

- 设置页已暴露 `approvalPolicy`、`sandboxSettings`，但治理能力必须以下沉到 Kernel/Runtime 为准，不能停留在 UI。
- 部分宿主与基础设施执行链仍处于标准化收口阶段；未满足统一权限、审计和证据链的能力不得进入正式发布链路。
- Tauri 可直接执行系统命令，必须统一纳入风险分级、审批策略、审计追踪。
- 当前 Terminal Runtime 已前移危险命令识别：`rm -rf`、`git reset --hard`、`Remove-Item -Recurse -Force` 等高风险命令在 `Restricted / ReleaseOnly` 下默认阻断。
- 当前阻断事件已同步写入本地终端治理审计存储，保证被拒绝的危险命令不仅能返回 `traceId`，还能被后续诊断和恢复流程回看。

## 3. 安全标准

- 身份分层：宿主身份、用户身份、引擎凭证三层分离。
- 权限最小化：文件、终端、网络、发布、Secrets 全部按最小权限控制。
- Secrets 不落日志、不进入前端持久化、不进入错误提示明文。
- `terminal.exec` 必须先做风险分级，再做审批判定；阻断结果也必须输出结构化审计事件，而不是仅做 UI 提示。
- 供应链对象必须可追溯：依赖、脚本、镜像、Chart、发布产物都要可验证来源。

- BirdCoder 进一步要求恢复入口直接消费同一批阻断审计：Terminal 下拉菜单必须展示最近的 `Governance Recovery` 记录，并复用统一恢复动作与说明文案，避免审计记录停留为死数据。
- 阻断审计不仅要可见，还要可复用：Terminal 必须能把当前可见的治理恢复记录导出为稳定 JSON 诊断包，保证排障、发布协作和后续归档不依赖手工整理 UI 文本。
- 发布协作同样要消费同一份治理事实：Terminal 必须能基于当前可见的治理恢复记录生成稳定 release-note markdown，避免发布说明再次手工转写阻断命令、策略和恢复结论。
## 4. 可观测性标准

- 日志至少包含 `traceId`、模块、动作、结果。
- 指标覆盖启动耗时、首响应耗时、构建时长、测试通过率、发布成功率。
- 链路追踪需覆盖用户动作、引擎执行、工具调用、工件生成。
- 审计必须覆盖权限变更、引擎切换、发布动作、危险操作。
- 当前终端阻断链已要求返回 `traceId`，保证危险命令被拒绝时也可追溯。
- 当前终端阻断链已把危险命令审计事件持久化到本地存储，保证“阻断结果”和“审计记录”是同一条事实链。

## 5. 稳定性标准

- 长任务统一支持取消、超时、重试、恢复。
- 任一 Adapter 故障不得拖垮整个 Shell。
- Release 流程必须可重入、可回滚、可补发证据。

## 6. 评估标准

- 性能指标是否可量化、可压测、可回归。
- 安全事件是否可阻断、可追溯、可告警。
- 审计数据是否足以定位问题，而不是只堆日志。
- 故障后是否存在明确恢复路径与责任边界。
