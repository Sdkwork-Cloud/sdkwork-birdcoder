# 10-开发流程-质量门禁-评估标准

## 1. 标准流程

1. 需求分析
2. 架构设计
3. 契约定义
4. 模块拆分
5. 实现与自测
6. 契约测试与 Smoke
7. 文档补齐
8. Release 与发布

## 2. 当前门禁基线

仓库已经冻结以下根命令：

- `typecheck`
- `check:source-parse`
- `check:arch`
- `check:sdkwork-birdcoder-structure`
- `check:quality:fast`
- `check:quality:standard`
- `check:quality:release`
- `check:quality-matrix`
- `check:data-kernel`
- `check:release-flow`
- `check:ci-flow`
- `check:governance-regression`
- `docs:build`

这代表 BirdCoder 已具备类型、数据内核、结构、架构、质量分层、发布、治理、文档八类门禁骨架。

## 3. 交付物要求

- 需求说明
- 架构与边界说明
- 接口契约
- 测试证据
- 发布说明
- 回滚与风险说明
- `releaseControl` 归档：`releaseKind`、`rolloutStage`、`monitoringWindowMinutes`、`rollbackRunbookRef`、`rollbackCommand`（可选）

## 4. 质量门禁

- 架构门禁：不允许破坏包边界、宿主隔离、appbase 单边界
- 契约门禁：对外接口必须版本化、可测试、可替换
- 代码门禁：关键路径必须可读、可测、可审计
- 文档门禁：新增能力必须同步更新架构、Step、Prompt、Release 文档
- 发布门禁：未完成 smoke、未完成 `releaseControl`、未完成 `release:rollback:plan` 或等价回滚入口不得发布

## 5. Review 维度

- 需求是否闭环
- 设计是否支持扩展与替换
- 功能是否跨引擎、跨宿主一致
- 性能与安全是否有定量标准
- 发布链路是否覆盖计划、打包、smoke、finalize、回滚

## 6. 评估分级

| 维度 | A | B | C |
| --- | --- | --- | --- |
| 架构 | 可扩展、无跨层耦合 | 轻微耦合可控 | 结构失衡 |
| 功能 | 主流程完整一致 | 局部降级可控 | 主流程缺失 |
| 性能 | 有指标且达标 | 有指标未完全达标 | 无指标 |
| 安全 | 边界清晰且有审计 | 有边界缺审计 | 边界模糊 |
| 发布 | `plan -> finalize -> notes -> GitHub` 全链路一致 | 半自动 | 人工拼接 |

## 7. 验收标准

- 至少达到 `B`，Kernel、Engine、Release 改动必须冲刺 `A`
- 发布链路必须能证明 `releaseControl`、制品、smoke、回滚入口一致

## 9. Current Loop Addendum - Executable Loop Scoreboard

- Finalized release quality evidence now materializes a deterministic `loopScoreboard`:
  - `architecture_alignment`
  - `implementation_completeness`
  - `test_closure`
  - `commercial_readiness`
  - `lowest_score_item`
  - `next_focus`
- The scoreboard is derived from executable quality evidence facts (`workflowBoundTiers`, release governance checks, execution-tier status, and blocking diagnostics), not from manual narrative scoring.
- `release-manifest.json.qualityEvidence`, finalized smoke verification, and rendered release notes now share the same loop-scoreboard summary, so the “lowest-score-item-first” loop rule is auditable in packaged release evidence.

- Correction: `architecture_alignment` is now derived from the lower of `workflowBoundTiers` and `manifestBoundTiers`, so manifest-topology drift cannot be hidden by workflow-only coverage.
- Correction: packaged `qualityEvidence` must preserve `missingWorkflowBindings` and `missingManifestBindings`, and rendered release notes must surface those tier ids whenever topology drift exists.
