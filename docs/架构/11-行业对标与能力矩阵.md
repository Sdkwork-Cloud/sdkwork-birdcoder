# 11-行业对标与能力矩阵

## 1. 对标方向

BirdCoder 不应单点模仿某一款工具，而应融合三类能力：

- 终端代理型：命令执行、仓库推理、工具编排。
- IDE 原生型：编辑器体验、多文件协同、即时反馈。
- 企业工程型：权限、审计、部署、发布、治理。

## 2. 核心对标对象

- Claude Code：终端优先、代理式编码、工具驱动工作流。
- Codex：统一任务执行、代码生成、测试与修复闭环。
- Gemini CLI：技能系统、工具注册、动态 instructions、会话上下文装配。
- Cursor / Windsurf：IDE 内联智能、规则系统、后台代理、交互效率。
- GitHub Copilot / JetBrains AI：企业 IDE 集成、多层规则、团队普及能力。
- OpenCode：开放式 Code Engine 接入、类型安全 SDK 与开放工具生态。

## 3. 行业启发点

- 顶级 AI IDE 都在把规则、工具、MCP、环境隔离和任务编排做成内建能力，而不是外挂脚本。
- 高效产品都在缩短“需求 -> 修改 -> 预览 -> 测试 -> 发布”的闭环时间。
- 企业级落地都要求权限、审计、发布证据和组织级规则注入。

## 4. 四大目标引擎给出的直接启发

| 引擎 | 从本地源码可确认的强项 | BirdCoder 必须继承的标准 |
| --- | --- | --- |
| Codex | `thread/turn/item/event` 清晰；CLI JSONL 与 app-server JSON-RPC v2 双入口；结构化输出明确 | 会话连续性、标准 turn 边界、结构化输出与宿主双通道 |
| Gemini | `sessionId`、Tool Registry、Skill 装载、动态 instructions、transcript 驱动 | 统一 ContextAssembler、SkillHub 注入、ToolBridge 编排 |
| OpenCode | OpenAPI 生成 SDK；`session/message/part` 清晰；`diff/todo/pty/question` 一等能力；自定义工具强 | 事件和 Artifact 不能只围绕文本消息建模，必须支持 IDE 原生产物 |
| Claude Code | remote-control、bridge、history、SSE、`tool_progress/result/stream_event` 适配清晰 | 本地/远程双宿主统一、审批与进度事件标准化、MCP 与远程控制一体化 |

## 5. BirdCoder 当前站位

- 强项：多宿主骨架、发布矩阵、release / smoke / GitHub 发布契约相对完整。
- 已冻结标准：共享 `types + kernel` 已固定 canonical engine key、descriptor、model catalog 与 `coding_session` 主命名。
- 当前成熟度：`coding-server` 代表性 `core / app / admin` 路由、canonical OpenAPI 发布证据与多引擎 canonical runtime 已闭环；Step 18 当前主线只在新增引擎接入或 fresh failing evidence 下重开。
- 当前主任务：把这些已闭环能力持续固化到 conformance、live-docs 与 release-governance 证据里，避免成熟度摘要回退。

## 6. BirdCoder 目标能力矩阵

| 维度 | 行业优秀做法 | BirdCoder 标准 |
| --- | --- | --- |
| 引擎接入 | 单工具深度绑定或少量适配 | 统一 SPI，多引擎一致体验 |
| IDE 体验 | 编辑器或终端局部突出 | 编辑器、终端、预览、构建一体化 |
| 工作流 | 偏聊天或偏补全 | 需求到发布全流程闭环 |
| 部署形态 | 单机或 SaaS 为主 | Web、Desktop、Server、Docker、K8s 全覆盖 |
| 治理能力 | 企业特性分散 | 权限、审计、配置、发布证据统一内建 |
| 扩展能力 | 插件化程度不一 | Engine、Tool、Preview、Build 全链路可插拔 |

## 7. 差异化要求

- 不做单一模型前端，而做多引擎统一开发内核。
- 不把预览、模拟、测试外包给零散工具链，而做成第一层产品能力。
- 不把发布理解为脚本集合，而做成标准化 release system。

## 8. 评估标准

- 是否已经形成“统一内核 + 统一 API + 统一体验”的产品壁垒。
- 是否可以在不重构业务层的前提下切换、扩展、替换底层引擎。
- 是否能够覆盖从编码到发布的完整专业开发链路。
- 与行业领先产品相比，是否在多宿主、多引擎和发布治理上形成明确优势。
