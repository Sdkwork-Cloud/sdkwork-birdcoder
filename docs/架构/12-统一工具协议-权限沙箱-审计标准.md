# 12-统一工具协议-权限沙箱-审计标准

## 1. 目标

BirdCoder 不只统一模型接口，还要统一工具协议。所有引擎必须通过同一套工具调用、审批、沙箱、审计规则访问 IDE 能力，避免页面直连文件、终端、发布能力。

## 2. 当前基线

- 当前工具能力已显性覆盖 `run_command`、`edit_file` 等主路径，说明统一协议雏形已经存在。
- 设置页已提供 `approvalPolicy`、`sandboxSettings`，但治理必须以下沉到 Runtime 为准。
- 终端已具备多 profile、多命令入口，Tauri 下通过 `@tauri-apps/plugin-shell` 执行真实系统 shell，Web 宿主仅允许通过统一 host-runtime adapter 暴露受限能力。
- 本轮已把终端治理前移到 `packages/sdkwork-birdcoder-commons/src/terminal/runtime.ts`，在命令真正执行前统一做风险判定和阻断。

## 3. 目标工具分类

- `fs.read` / `fs.write` / `fs.rename` / `fs.delete`
- `terminal.exec`
- `git.status` / `git.diff` / `git.commit`
- `search.code` / `search.symbol`
- `build.run` / `test.run` / `preview.open`
- `release.package` / `release.smoke` / `release.finalize`
- `mcp.call`
- `secret.request`

## 4. 权限分级

- `P0`：只读，无副作用。
- `P1`：可写，但局部、可回滚。
- `P2`：触发外部执行、子进程或环境变化。
- `P3`：发布、部署、密钥读取、危险删除、不可逆操作。

## 5. 审批策略

- `AutoAllow`：仅允许 `P0`。
- `OnRequest`：`P1/P2` 需审批。
- `Restricted`：`P2/P3` 默认拒绝。
- `ReleaseOnly`：发布链路独立审批，不复用开发审批。

## 6. 沙箱标准

- 文件沙箱：限制工作区范围。
- 命令沙箱：限制可执行命令、`cwd`、环境变量。
- 网络沙箱：限制外连域名与协议。
- 发布沙箱：限制目标仓库、镜像仓库、命名空间。

## 7. 审计标准

每次工具调用至少记录：

- `traceId`
- `engine`
- `tool`
- `riskLevel`
- `approvalDecision`
- `inputDigest`
- `outputDigest`
- `artifactRefs`
- `operator`

上述审计字段、`P0-P3` 风险等级与 `AutoAllow / OnRequest / Restricted / ReleaseOnly` 审批策略已冻结到 `packages/sdkwork-birdcoder-types/src/governance.ts`。`terminal.exec` 的风险分级、阻断返回、审计类别与 `traceId` 语义现统一由共享治理模型与发布证据链路承载，不再维护独立的 terminal governance 契约测试。

- `terminal.exec` 的危险命令阻断事件会写入 `packages/sdkwork-birdcoder-commons/src/terminal/auditStore.ts`，把 `traceId / command / cwd / approvalPolicy / category / reason` 沉淀为统一审计记录。
- `resolveTerminalGovernanceRecoveryAction`、`buildTerminalGovernanceRecoveryDescription`、`buildTerminalGovernanceDiagnosticBundle(...)`、`buildTerminalGovernanceReleaseNoteTemplate(...)` 共同冻结恢复、诊断、发布协作的统一治理语义；`TerminalPage` 只消费这些共享结果，不自定义平行逻辑。
- `scripts/release/finalize-release-assets.mjs`、`scripts/release/smoke-finalized-release-assets.mjs`、`scripts/release/render-release-notes.mjs` 现在统一以 `qualityEvidence`、desktop startup readiness 与 Studio evidence 作为发布事实源，不再额外挂接旧 terminal governance 归档链路。
- `scripts/governance-regression-report.mjs` 当前聚合 100 项既有合同检查，并已纳入 `package-governance`、`release-rollback-plan-command`、`quality-gate-matrix`、`quality-loop-scoreboard`、host-runtime、host-studio、Studio execution/evidence、official-SDK-first engine governance、release closure。它是统一聚合资产，不再额外定义第二套治理语义。

## 8. 当前关键差距

- 仍缺独立 Tool Registry、Tool Bridge 与统一执行网关。
- 更多工具类型仍停留在目标协议层，尚未形成完整实现矩阵。
- 终端、AI tool call、发布流程、MCP 调用尚未完全共享同一条审批与审计主线。
- 终端治理证据已进入 finalized manifest 和 release notes，但跨更多工具族的统一归档接口仍待继续扩展。
- 虽然已有统一 governance regression 报告，且已扩到 host-runtime、host-studio 与 Studio 执行主线，但更多工具类型与宿主模式尚未纳入这条可重复回归链。
- Tauri 真执行与 Web 受限宿主执行的行为差异还需继续收口到统一能力模型。

## 9. 评估标准

- 同一工具在不同引擎、不同宿主下语义一致。
- 高风险操作一定经过审批、审计、可回放。
- 任一副作用都能追溯到任务、引擎、操作者、工具和目标环境。
- 页面移除后，工具协议仍可由 Kernel 独立运行。
