## 3. 主要差距

- `coding-server` 协议标准已冻结，但 Rust host 仍是最小骨架，`core / app / admin` 真实服务面尚未落地。
- `useProjects.sendMessage()` 仍保留 Hook 层结果归一；真实 turn orchestration、event projection、artifact projection 还未完全下沉到 Kernel / server。
- 多引擎适配当前以协议标准和本地源码镜像为主，真实 CLI/SDK/remote-control adapter 还未形成统一生产级执行栈。
- 终端治理已覆盖 `terminal.exec` 并进入 release 资产链，但更多工具类型与跨宿主统一治理归档仍未落地。
- 当前标准 / 发布门禁仍受宿主 `esbuild` 二进制子进程能力约束；该问题已被 `run-vite-host` preflight 明确暴露为 `toolchain-platform` 前提缺失，不应再当作 BirdCoder 架构漂移处理。
- 真实 Secrets、网络沙箱、发布沙箱仍需从配置层继续下沉到内核执行层。
- 性能、安全、审计门禁还需继续扩展到 server、desktop、release 产物校验。

## 4. 演进路线

1. Terminal 治理收口
   - 继续补齐审批恢复、人工放行、证据归档、发布链复用，并向跨工具统一治理归档推进。
2. `coding-server` 实装
   - 先落地 `core / app / admin` 路由、鉴权、SSE、OpenAPI，再把桌面和 Web 调用收口到同一协议面。
3. Turn 与 Artifact 下沉
   - 把 `useProjects.sendMessage()` 中的结果归一迁回 Kernel / server，统一 turn orchestration、event projection、artifact projection。
4. Tool 协议扩面
   - 把 `fs.*`、`build.run`、`test.run`、`release.*`、`mcp.call` 接入同一治理主线。
5. Release 与诊断闭环
   - 把 Step 07 证据视图、Step 10 审计事件、Step 12 质量证据汇聚到统一诊断和发布资产。

## 5. 评估标准

| 维度 | 目标 |
| --- | --- |
| 架构一致性 | 多 lane、多引擎、多宿主共享同一治理合同和执行语义 |
| 工程门禁 | 每条能力都有脚本化合同测试并进入 `lint` 或 release-flow |
| 交付效率 | 新增 lane 只新增适配器和合同，不重造执行主链 |
| 商业化准备度 | 证据、审计、发布、回放、恢复链条可支撑桌面版、Server、Docker、K8s 发布 |
